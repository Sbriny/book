### 一、行业应用开发及技术架构

### 二、名词解释

#### GIS

**Geographic Information System / 地理信息系统**

用于采集、存储、处理、分析和展示地理空间数据的系统。不是单一软件，而是一个技术体系的统称。涵盖从卫星采集数据到最终在地图上展示的全部环节。

> GIS vs Geo
>
> **GIS** — 整个技术体系和学科领域的名称。
>
> **Geo** — 前缀，来自希腊语"地球"，在技术语境中通常是"地理相关"的缩写，如 GeoJSON、Geocoding、Geospatial。不是一个独立的技术名词。

#### GeoJSON

用 JSON 格式描述地理要素（点、线、面）的开放标准（RFC 7946）。是 Web 地图开发中最常用的地理数据格式。

```text
{
  "type": "Feature",
  "geometry": {
    "type": "Point",
    "coordinates": [116.4074, 39.9042]
  },
  "properties": {
    "name": "北京"
  }
}
```

三种核心几何类型：`Point`（点）、`LineString`（线）、`Polygon`（面）。

## TopoJSON

**TopoJSON**

GeoJSON 的压缩版本，通过共享边界减少冗余数据，文件体积可减少 80% 以上。代价是不能直接被地图引擎读取，需要先转换为 GeoJSON。ECharts 世界地图加载失败的根本原因就是把 TopoJSON 当 GeoJSON 用。

## Shapefile

**Shapefile（.shp）**

Esri 公司制定的地理数据格式，GIS 专业领域的历史标准格式。一个"文件"实际上由多个文件组成（.shp / .dbf / .prj 等）。Web 开发中通常需要用 GDAL 转换为 GeoJSON 才能使用。

## 瓦片 / Tile

**Map Tile / 地图瓦片**

将地图切割成固定尺寸（通常 256×256 像素）的小图片，按照缩放级别（Zoom Level）组织。用户看到的地图是由数十到数百张瓦片拼合而成的。

```text
Zoom 0  →  整个世界 = 1 张瓦片
Zoom 1  →  整个世界 = 4 张瓦片
Zoom 18 →  街道级别，瓦片数量以亿计
```

分为两种：**栅格瓦片（Raster Tile）** — PNG/JPEG 图片，放大会模糊；**矢量瓦片（Vector Tile）** — 包含原始几何数据，可以在客户端自由渲染样式，放大不失真。

## 底图 / Basemap

**Basemap / 底图**

作为地理参考背景的地图层，通常包含道路、建筑、水系、地名等基础地理信息。数据可视化内容叠加在底图上方。常见底图风格有街道图、卫星图、暗色科技风、地形图等。

## 地图引擎 / Map Engine

**Map Rendering Engine / 地图渲染引擎**

负责加载瓦片数据并在浏览器中渲染成可交互地图的库。核心职责是处理缩放、平移、旋转、坐标投影等地图基础交互。代表：MapLibre GL JS、Leaflet、CesiumJS、OpenLayers。

## 数据可视化工具 / Data Visualization Library

**Geospatial Data Visualization Library / 地理数据可视化库**

在地图引擎提供的地理坐标系上，将业务数据渲染成散点、热力图、弧线、3D 柱体等视觉形式的库。不负责画地图本身，只负责画数据。代表：[deck.gl](http://deck.gl)、ECharts、D3.js、[kepler.gl](http://kepler.gl)。

## 坐标系 / Coordinate System / CRS

**Coordinate Reference System / 坐标参考系统**

定义如何用数字描述地球上某个位置的规则体系。不同坐标系对地球形状的数学建模方式不同，同一个物理位置在不同坐标系下会有不同的经纬度数值。

## WGS-84

**World Geodetic System 1984 / 世界大地坐标系 1984**

GPS 卫星系统使用的国际标准坐标系，是全球通用的基准。所有 GPS 硬件输出、OpenStreetMap 数据、Google Maps 境外数据均使用此坐标系。经纬度即为地球上的真实位置。

## GCJ-02

**国测局坐标系 / 火星坐标系**

中国国家测绘局制定的坐标系，在 WGS-84 基础上叠加了一个非线性偏移算法。中国境内发布的地图（高德、腾讯、Google 中国区道路层）必须使用此坐标系。偏移量约为 100 ~ 700 米，算法已被逆向破解但官方从未公开。

## BD-09

**百度坐标系**

百度地图在 GCJ-02 基础上再次叠加偏移的私有坐标系。仅百度地图使用，与其他坐标系均不兼容，需要单独转换。

## CGCS2000

**China Geodetic Coordinate System 2000 / 2000 国家大地坐标系**

中国北斗卫星导航系统使用的官方坐标系。与 WGS-84 差距在厘米级，工程实践中通常视为等同处理。

## Geocoding / 地理编码

**Geocoding / 地理编码**

将地址文字转换为经纬度坐标的过程。例如"北京市朝阳区三里屯"→ `[116.45, 39.93]`。反向操作（坐标转地址）称为 **Reverse Geocoding / 逆地理编码**。

## 地理围栏 / Geofence

**Geofence / 地理围栏**

在地图上定义一个虚拟边界区域，当设备进入或离开该区域时触发事件。常用于外卖配送范围判断、车辆进出厂区提醒等场景。

## PostGIS

**PostGIS**

PostgreSQL 数据库的地理空间扩展，使普通关系型数据库具备存储和查询地理数据的能力。可以直接用 SQL 执行"找出距离某点 5 公里内的所有记录"这类空间查询。

## GDAL

**Geospatial Data Abstraction Library / 地理空间数据抽象库**

处理地理数据格式转换的开源库，支持 200\+ 种地理数据格式的读写和转换。命令行工具 `ogr2ogr` 基于 GDAL，常用于 Shapefile → GeoJSON 的转换。

## Turf.js

**Turf.js**

在浏览器或 Node.js 中进行空间计算的 JavaScript 库，无需后端即可执行缓冲区分析、面积计算、点是否在多边形内等空间运算。

## PMTiles

**PMTiles**

一种单文件的地图瓦片存储格式，可以直接存放在对象存储（S3/OSS）上对外提供服务，不需要运行专门的瓦片服务器。是自建地图服务的现代化低成本方案。

## WebGL

**Web Graphics Library**

浏览器内置的 3D 图形 API，允许直接调用 GPU 渲染图形。MapLibre、[deck.gl](http://deck.gl) 等现代地图引擎均基于 WebGL 渲染，这是它们能流畅处理大量数据点的根本原因。

## DEM

**Digital Elevation Model / 数字高程模型**

记录地表每个点海拔高度的栅格数据。CesiumJS 和 Mapbox 的 3D 地形效果依赖 DEM 数据实现。

## RTK

**Real-Time Kinematic / 实时动态差分定位**

通过地面基准站实时修正 GPS 误差的技术，精度可达厘米级。测量仪器、无人机精准降落、自动驾驶辅助定位使用此技术。

# 三、辨析及注意事项

## 地图引擎 vs 数据可视化工具

**最常见的混淆点。**

|  | 地图引擎 | 数据可视化工具 |
| :-- | :-- | :-- |
| 职责 | 画地图本身 | 把数据画在地图上 |
| 代表 | MapLibre、Leaflet | [deck.gl](http://deck.gl)、ECharts |
| 能否独立使用 | 能，显示底图即可 | 通常需要依附地图引擎 |
| 渲染对象 | 瓦片、道路、建筑 | 散点、热力、弧线 |

两者是搭档关系，不是竞争关系。ECharts 是例外——它内置了简单的世界地图 GeoJSON，可以不依赖地图引擎独立运行，但功能受限。

## GeoJSON vs TopoJSON vs Shapefile

| 格式 | 场景 | 能否直接用于 Web |
| :-- | :-- | :-- |
| GeoJSON | Web 开发首选 | 能 |
| TopoJSON | 文件压缩、传输优化 | 需先转换 |
| Shapefile | GIS 专业软件、历史数据 | 需先转换 |

**注意**：TopoJSON 文件开头是 `{"type":"Topology"...}`，GeoJSON 开头是 `{"type":"FeatureCollection"...}`，两者长得像但不兼容，这是很多新手踩的第一个坑。

## WGS-84 vs GCJ-02 vs BD-09

```text
WGS-84  →  GCJ-02  →  BD-09
  GPS      高德/腾讯    百度
```

转换只能单向推导，逆向转换需要迭代近似算法，不能做到完全精确。

**实际开发注意事项：**

国内 iOS 系统定位 API 返回的已经是 GCJ-02，不需要再转。境外 GPS 设备、相机 EXIF 坐标、OSM 数据是 WGS-84，叠加在高德底图上会偏移，必须转换。混用不同来源坐标是国内地图开发最常见的 bug 来源。

## 栅格瓦片 vs 矢量瓦片

|  | 栅格瓦片 | 矢量瓦片 |
| :-- | :-- | :-- |
| 格式 | PNG / JPEG 图片 | PBF / MVT 二进制 |
| 放大效果 | 模糊 | 清晰 |
| 样式修改 | 不能，图片已烘焙 | 能，客户端实时渲染 |
| 深浅色切换 | 需要两套瓦片 | 一套数据切换样式即可 |
| 流量 | 较大 | 较小 |
| 兼容性 | 所有引擎支持 | 需要 GL 引擎支持 |

你 Demo 里 CARTO 用的就是矢量瓦片，所以能做深浅色主题切换。

## 地图引擎 vs GIS 平台

|  | 地图引擎 | GIS 平台 |
| :-- | :-- | :-- |
| 代表 | MapLibre、Leaflet | ArcGIS、QGIS |
| 使用场景 | Web 应用开发 | 专业地理分析 |
| 使用者 | 前端/全栈开发者 | GIS 专业人员 |
| 分析能力 | 基础或无 | 完整空间分析 |
| 学习成本 | 低到中 | 高 |

## GPS 精度 vs 坐标系偏移

两个完全不同的概念，经常被混淆。

**GPS 精度误差** — 物理层面，信号传播造成的定位不准，手机级别 3～50 米，无法消除只能减小。

**坐标系偏移** — 软件层面，人为的数学变换，在中国境内固定偏移约 100～700 米，可以通过算法完全补偿。

前者是物理问题，后者是政策问题，解决方式完全不同。

## [deck.gl](http://deck.gl) vs [kepler.gl](http://kepler.gl)

|  | [deck.gl](http://deck.gl) | [kepler.gl](http://kepler.gl) |
| :-- | :-- | :-- |
| 定位 | 开发者框架 | 完整产品 |
| 使用方式 | 写代码 | 拖拽 UI |
| 底层 | 原生 WebGL | 基于 [deck.gl](http://deck.gl) |
| 适合 | 自定义开发 | 快速探索数据 |

[kepler.gl](http://kepler.gl) 是基于 [deck.gl](http://deck.gl) 构建的，两者出自同一团队（Uber Visualization）。

## Zoom Level / 缩放级别

地图缩放不是线性的，每增加 1 级，瓦片数量变为 4 倍，精度翻倍。

```text
Zoom 0   →  全球视野
Zoom 5   →  大洲/国家级
Zoom 10  →  城市级
Zoom 15  →  街道级
Zoom 18  →  建筑级（大多数底图最高级别）
Zoom 22  →  室内级（高精地图）
```
